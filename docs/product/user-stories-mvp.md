## Epics

MVP охватывает Фазу 1 (Backend) и Фазу 2 (Apple Watch + iOS companion). Выделены следующие эпики:

| # | Epic | Описание | Компонент |
|---|------|----------|-----------|
| E1 | Аутентификация и управление пользователями | Регистрация, логин, JWT-токены | Backend |
| E2 | Управление API-ключами провайдеров | Добавление, валидация, шифрование и хранение read-only ключей | Backend |
| E3 | Сбор данных от LLM-провайдеров | Интеграция с Anthropic, OpenAI, Vertex AI; polling, кэширование | Backend |
| E4 | REST API для клиентов | Эндпоинты summary, limits, usage, costs | Backend |
| E5 | iOS Companion App | Настройка ключей, просмотр дашборда, синхронизация с часами | iOS |
| E6 | watchOS App | Экраны summary и detail, обновление данных на часах | watchOS |
| E7 | Push-уведомления | Оповещения при приближении к лимитам (>80%, >95%) | Backend + iOS + watchOS |

## Acceptance Criteria

Критерии приёмки описаны для всех Must-have stories.

### US-01: Регистрация пользователя

```gherkin
Given пользователь не зарегистрирован в системе
When он отправляет POST /auth/register с валидным email и паролем (>=8 символов)
Then система создаёт аккаунт и возвращает 201 с JWT-токенами (access + refresh)

Given пользователь отправляет POST /auth/register с уже существующим email
When запрос обрабатывается
Then система возвращает 409 Conflict с сообщением "Email already registered"

Given пользователь отправляет POST /auth/register с невалидным email или коротким паролем
When запрос обрабатывается
Then система возвращает 422 Validation Error с описанием ошибки
```

### US-02: Логин пользователя

```gherkin
Given пользователь зарегистрирован в системе
When он отправляет POST /auth/login с корректными email и паролем
Then система возвращает 200 с JWT access-токеном и refresh-токеном

Given пользователь отправляет POST /auth/login с неверным паролем
When запрос обрабатывается
Then система возвращает 401 Unauthorized

Given пользователь обращается к защищённому эндпоинту без токена или с истёкшим токеном
When запрос обрабатывается
Then система возвращает 401 Unauthorized
```

### US-04: Добавление ключа Anthropic

```gherkin
Given пользователь аутентифицирован
When он отправляет POST /auth/providers с типом "anthropic", валидным read-only Admin API key и указанным tier
Then система валидирует ключ (тестовый запрос к usage endpoint), шифрует его и сохраняет вместе с tier; возвращает 201

Given пользователь не указал tier при добавлении ключа Anthropic
When запрос обрабатывается
Then возвращается 422 Validation Error с сообщением "Укажите тарифный план Anthropic для расчёта rate limits"

Given пользователь отправляет ключ, который не является read-only (позволяет inference)
When система проводит валидацию
Then возвращается 400 Bad Request с сообщением "Key is not read-only"

Given пользователь отправляет невалидный или отозванный ключ
When система проводит валидацию
Then возвращается 400 Bad Request с сообщением "Key validation failed"
```

> **Примечание:** Tier нужен потому, что Anthropic Admin API не предоставляет endpoint для rate limits.
> Read-only ключ не может получить заголовки `anthropic-ratelimit-*` (они доступны только через Messages API).
> RPM/TPM `used` рассчитывается из usage report (`bucket_width=1m`), а `limit` берётся из статической таблицы по tier.

### US-05: Добавление ключа OpenAI

```gherkin
Given пользователь аутентифицирован
When он отправляет POST /auth/providers с типом "openai" и валидным read-only Admin API key
Then система валидирует ключ (тестовый запрос к usage endpoint), шифрует и сохраняет; возвращает 201

Given ключ невалиден или не read-only
When система проводит валидацию
Then возвращается 400 Bad Request с описанием ошибки
```

### US-06: Добавление Service Account JSON для Vertex AI

```gherkin
Given пользователь аутентифицирован
When он отправляет POST /auth/providers с типом "vertex_ai" и валидным Service Account JSON с viewer-ролями
Then система валидирует SA (тестовый запрос к Monitoring API), шифрует JSON и сохраняет; возвращает 201

Given Service Account не имеет необходимых viewer-ролей
When система проводит валидацию
Then возвращается 403 с сообщением "Insufficient permissions: requires monitoring.viewer, billing.viewer"
```

### US-07: Валидация read-only при добавлении ключа

```gherkin
Given пользователь добавляет ключ любого провайдера
When система проводит валидацию
Then выполняется: 1) проверка формата ключа, 2) тестовый read-запрос, 3) проверка что inference-запрос отклоняется
  And результат валидации возвращается пользователю

Given ключ проходит все проверки
When ключ сохраняется в БД
Then он зашифрован с помощью Fernet (AES-256) и никогда не хранится в открытом виде
```

### US-09: Периодический polling провайдеров

```gherkin
Given у пользователя добавлен хотя бы один провайдер
When проходит интервал обновления (1-5 мин)
Then бэкенд выполняет фоновый запрос к API провайдера и обновляет кэш

Given провайдер не отвечает или возвращает ошибку
When происходит polling
Then система использует последние закэшированные данные и помечает провайдер статусом "stale"
```

### US-10: Кэширование ответов

```gherkin
Given клиент запрашивает GET /api/v1/summary
When данные по провайдеру были получены менее 60 секунд назад (rate limits) или менее 5 минут назад (usage/costs)
Then система возвращает закэшированные данные без обращения к провайдеру

Given кэш истёк
When клиент запрашивает данные
Then система запрашивает свежие данные у провайдера, обновляет кэш и возвращает результат
```

### US-12: Эндпоинт GET /api/v1/summary

```gherkin
Given пользователь аутентифицирован и имеет добавленные провайдеры
When он запрашивает GET /api/v1/summary
Then система возвращает JSON со списком провайдеров, каждый содержит: id, name, status, rpm (used/limit/pct), tpm (used/limit/pct), cost_today, cost_month, budget_month, budget_pct, updated_at

Given у пользователя нет ни одного провайдера
When он запрашивает GET /api/v1/summary
Then система возвращает 200 с пустым списком providers и подсказкой добавить ключ
```

### US-15: Ввод ключей в iOS companion app

```gherkin
Given пользователь открывает экран настроек в iOS-приложении
When он выбирает провайдера (Anthropic/OpenAI/Vertex AI)
Then отображается пошаговая инструкция по созданию read-only ключа и поле для ввода

Given пользователь вводит ключ и нажимает "Сохранить"
When ключ отправляется на бэкенд
Then отображается результат валидации (успех или ошибка с описанием)
```

### US-16: Дашборд в iOS-приложении

```gherkin
Given пользователь аутентифицирован и имеет добавленные провайдеры
When он открывает главный экран iOS-приложения
Then отображается список провайдеров с индикаторами статуса и ключевыми метриками (RPM%, TPM%, расходы)

Given данные загружаются
When экран открыт
Then отображается скелетон/индикатор загрузки; при ошибке — сообщение с кнопкой "Повторить"
```

### US-17: Синхронизация с Apple Watch через WatchConnectivity

```gherkin
Given пользователь аутентифицирован в iOS-приложении
When iOS-приложение получает или обновляет JWT-токен
Then токен передаётся на Apple Watch через WatchConnectivity и сохраняется в Keychain часов

Given Apple Watch получил JWT
When пользователь открывает приложение на часах
Then часы могут самостоятельно обращаться к бэкенду для получения данных
```

### US-19: Summary-экран на Apple Watch

```gherkin
Given приложение на часах имеет валидный JWT и данные от бэкенда
When пользователь открывает приложение
Then отображается список провайдеров с цветовыми индикаторами:
  - Зелёный: все метрики < 60%
  - Жёлтый: хотя бы одна метрика 60-85%
  - Красный: хотя бы одна метрика > 85%

Given нет данных или JWT невалиден
When пользователь открывает приложение на часах
Then отображается сообщение "Откройте приложение на iPhone для настройки"
```

### US-20: Детальный экран провайдера на Apple Watch

```gherkin
Given пользователь находится на summary-экране
When он нажимает на провайдера
Then открывается экран с круговыми прогресс-барами: RPM% (использовано/лимит), TPM% (использовано/лимит), бюджет% (потрачено/бюджет), стоимость за сегодня

Given данные устарели (статус "stale")
When отображается детальный экран
Then индикатор времени обновления отображается серым с пометкой "Обновлено X мин назад"
```

### US-22: Хранение JWT в Keychain на Apple Watch

```gherkin
Given JWT-токен передан на Apple Watch
When токен сохраняется
Then он записывается в Keychain с атрибутом kSecAttrAccessible = kSecAttrAccessibleWhenUnlocked

Given Keychain недоступен или токен истёк
When приложение на часах пытается обратиться к API
Then отображается сообщение о необходимости повторной аутентификации через iPhone
```

## Риски и зависимости

### Риски

| # | Риск | Вероятность | Влияние | Митигация |
|---|------|-------------|---------|-----------|
| R1 | Провайдеры изменят формат API или отзовут Admin API | Средняя | Высокое | Абстрактный интерфейс провайдера (AbstractProvider); мониторинг изменений API; версионирование адаптеров |
| R2 | Anthropic Admin API или OpenAI Admin API могут быть недоступны для некоторых тарифов | Средняя | Высокое | Проверка доступности при онбординге; понятное сообщение об ошибке; документация необходимых тарифов |
| R3 | Ограничения watchOS на фоновые задачи (Background App Refresh не гарантирован) | Высокая | Среднее | Обновление при открытии приложения; pull-to-refresh; информирование пользователя о возрасте данных |
| R4 | Пользователь по ошибке добавит ключ с write-правами | Низкая | Высокое | Обязательная валидация read-only при добавлении; тестовый inference-запрос должен завершиться ошибкой |
| R5 | Утечка зашифрованных ключей из БД | Низкая | Критическое | Fernet-шифрование; ротация Fernet-ключа; минимальные права доступа к БД; аудит доступов |
| R6 | Apple может отклонить приложение в App Store Review | Средняя | Высокое | Соблюдение Human Interface Guidelines; наличие iOS companion app; понятная функциональность |
| R7 | Rate limiting со стороны провайдеров при массовом polling | Средняя | Среднее | Адаптивные интервалы polling; экспоненциальный backoff; кэширование |

### Зависимости

| # | Зависимость | Тип | Влияние |
|---|------------|-----|---------|
| D1 | Anthropic Admin API (usage_report, cost_report) | Внешняя | Без неё не работает мониторинг Claude |
| D2 | OpenAI Admin API (usage, costs, rate_limits) | Внешняя | Без неё не работает мониторинг GPT |
| D3 | Google Cloud APIs (Monitoring, Billing, Service Usage) | Внешняя | Без них не работает мониторинг Vertex AI |
| D4 | Apple Developer Program membership | Внешняя | Необходимо для тестирования на реальных устройствах и публикации |
| D5 | APNs (Apple Push Notification service) | Внешняя | Необходимо для push-уведомлений (Could) |
| D6 | Хостинг (Fly.io / Railway) | Инфраструктурная | Бэкенд должен быть доступен 24/7 для polling и обслуживания запросов |

## Definition of Done

MVP считается завершённым, когда выполнены все следующие критерии:

### Backend
- [ ] Эндпоинты регистрации и логина работают, возвращают JWT
- [ ] Можно добавить read-only ключи для Anthropic, OpenAI и Vertex AI с валидацией
- [ ] Ключи хранятся в БД в зашифрованном виде (Fernet)
- [ ] Фоновый polling провайдеров работает с заданными интервалами
- [ ] Кэширование ответов работает (rate limits — 60 сек, usage/costs — 5 мин)
- [ ] `GET /api/v1/summary` возвращает корректную сводку по всем провайдерам пользователя
- [ ] Ошибки одного провайдера не влияют на отображение других
- [ ] Unit-тесты покрывают провайдеры (mock API) и ключевые эндпоинты (>=80% coverage)
- [ ] Бэкенд задеплоен и доступен по HTTPS

### iOS Companion App
- [ ] Экран авторизации (регистрация + логин) работает
- [ ] Экран настроек позволяет добавить ключи для 3 провайдеров с инструкциями
- [ ] Дашборд отображает сводку по провайдерам с индикаторами статуса
- [ ] JWT и настройки передаются на Apple Watch через WatchConnectivity
- [ ] JWT хранится в Keychain

### watchOS App
- [ ] Summary-экран отображает список провайдеров с цветовыми индикаторами
- [ ] Детальный экран показывает круговые прогресс-бары (RPM%, TPM%, бюджет%)
- [ ] Часы самостоятельно запрашивают данные с бэкенда по полученному JWT
- [ ] Background App Refresh настроен для периодического обновления
- [ ] JWT хранится в Keychain часов
- [ ] Приложение корректно обрабатывает отсутствие данных и ошибки сети

### Общее
- [ ] Код в git-репозитории, CI проходит
- [ ] Нет хранения секретов в открытом виде (ключи, пароли)
- [ ] Приложение протестировано на watchOS симуляторе и реальном устройстве
- [ ] Все Must-have user stories реализованы и приняты по acceptance criteria
